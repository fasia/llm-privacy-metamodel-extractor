# Prompt Architecture — Privacy Policy Extraction Engine

## Overview

This document explains how the metamodel drives the LLM prompting strategy.
The central design principle is that **the schema IS the prompt** — every
extraction target, every enum constraint, and every OCL rule in the metamodel
has a direct, traceable counterpart in the prompt system.

---

## The Three-Layer Mapping

```
PrivacyPolicyMetamodel
        │
        ▼
Pydantic Schema (models.py / enums.py)
        │
        ▼
Prompt Templates (prompts.py)
        │
        ▼
LLM Output → Validation → Model Repository
```

Nothing in the prompts is invented independently of the metamodel.
If a field exists in the metamodel, it appears in the prompt.
If a field is removed from the metamodel, the prompt loses it automatically
because `build_concept_prompt()` derives enum grammars from `_ENUM_GRAMMARS`,
which mirrors `ControlledVocabularies` in the metamodel JSON.

## Prompt Anatomy — LegalBasis as the Worked Example

Every concept prompt has six sections. Here is what each section does and
why it exists, using `LegalBasis` as the concrete case.

### Section 1 — Task header

```
## Task: Extract a LegalBasis instance
```

Short, unambiguous. The LLM knows exactly which metamodel class it is filling.
One call = one class = one JSON object returned.

### Section 2 — Cross-law mapping

```
Maps to: GDPR Art.6(1)(a-f) | LGPD Art.7 | CCPA business-purpose | PIPEDA Sch.1
```

This is the **reuse mechanism** for the metamodel. The same `LegalBasis` class
captures the lawful-basis concept across all laws. The LLM needs to know which
legal article in each law maps to this abstract concept so that RAG retrieval
targets the right chunks. Without this, the model would not know that LGPD
Art.7(I) is the same concept as GDPR Art.6(1)(a).

### Section 3 — Output schema

The JSON skeleton the LLM must fill. Field names use the **alias** names from
the Pydantic model (e.g. `basisId` not `basis_id`) so the output can be fed
directly into `model_validate()` without any key transformation.

Crucially, the schema is **annotated with multiplicity**:
- `"jurisdiction": [{ ... }]`  — the list wrapper signals `1..*`
- `"riskAssessmentReference": null`  — the null default signals `0..1`
- `"dataProcessed": [{ ... }]` — required list signals `1..*`

The LLM reads the schema shape and infers the cardinality rules from it,
reinforced by the explicit OCL hints in Section 5.

### Section 4 — Enum grammar

```
LegalBasisType: Consent | Contract | LegalObligation | LegitimateInterest | VitalInterest | PublicTask
```

This is the **output vocabulary constraint**. The LLM may only output one of
these values for the `type` field. This directly mirrors the `LegalBasisType`
enum in `ControlledVocabularies`. Any other value is a Pydantic validation
error and the extraction is rejected.

The grammar block is generated by `_enum_block()` at call time from
`_ENUM_GRAMMARS`, which is kept in sync with `enums.py`.

### Section 5 — Decision guide and OCL constraint hints

The **decision guide** maps surface language in legal text to enum values:

| Text contains...          | `type`             |
|---------------------------|--------------------|
| consent / opts in         | `Consent`          |
| contract / service terms  | `Contract`         |
| legal obligation          | `LegalObligation`  |
| legitimate interest       | `LegitimateInterest` |

This handles cross-law variation. GDPR says "consent". LGPD says "consentimento".
A CCPA "business purpose disclosure" maps to `LegitimateInterest`. The guide
encodes this mapping so the LLM does not need to infer it.

**OCL constraint hints** translate the metamodel's formal constraints into
natural-language instructions the LLM acts on *before* outputting:

| OCL expression                                        | Prompt instruction                              |
|-------------------------------------------------------|-------------------------------------------------|
| `evidence->notEmpty()`                               | "evidence is mandatory — must quote the text"   |
| `jurisdiction` multiplicity `1..*`                   | "include at least one jurisdiction object"      |
| `mechanism=AdequacyDecision implies ref->notEmpty()` | "if mechanism=AdequacyDecision, cite the doc"  |

This is the **pre-validation layer**. The LLM corrects itself before output;
Pydantic enforces it after. Two chances to catch errors means fewer rejections.

### Section 6 — Few-shot example

One concrete input → output pair using real legal text. The example uses the
**same article** targeted by the prompt (GDPR Art.6(1)(a) for LegalBasis) so
the LLM sees the expected format in the exact context it will encounter.

The example output uses the **alias field names** (`basisId` not `basis_id`),
empty strings for IDs, and a populated `source_clause` — reinforcing all three
of the system-prompt rules at once.

## Two-Pass Extraction Strategy

### Why two passes?

A `PolicyStatement` has 12 navigable associations. Extracting all of them in a
single prompt would require injecting the entire statute into one context window,
which is both expensive and imprecise — the model attends to everything equally
instead of focusing on the relevant article.

The two-pass strategy keeps each call focused:

```
Pass 1: one API call per concept
        LegalBasis      ← Art.6 chunks from RAG
        RetentionPolicy ← Art.5(1)(e) chunks from RAG
        Rights          ← Art.17-22 chunks from RAG
        ConsentWithdrawal ← Art.7(3) chunks from RAG
        ...

Pass 2: one assembler call
        Takes all Pass 1 validated objects as structured JSON context
        Composes them into a single PolicyStatement
        Re-runs consistency checks (constraint_2, constraint_3, constraint_4)
        Returns the assembled PolicyStatement JSON
```

### Pass 1 — concept extraction

`build_concept_prompt(concept, law_name, article_ref, legal_text)` generates
the prompt for one concept. The RAG layer controls `legal_text` — it retrieves
only the chunks most relevant to that concept for that law, typically 1-3 clauses.

### Pass 2 — assembly

`build_assembler_prompt(...)` takes the Pass 1 outputs as JSON strings and
asks the LLM to compose them into a `PolicyStatement`. The assembler does NOT
re-read the legal text. It only sees the structured objects.

The assembler also re-runs the three OCL warnings in natural language:
- `constraint_2`: every regulation must have a non-empty jurisdiction
- `constraint_3`: high-risk Transfer/Share needs a risk assessment reference
- `constraint_4`: Consent legal basis requires ConsentWithdrawal mechanics

If the LLM flags any of these in a `"_warnings"` list, the pipeline logs them
as compliance gaps before the final Pydantic validation step.

## Validation Pipeline

```
LLM raw JSON output
        │
        ▼
json.loads()                    ← reject if not parseable JSON
        │
        ▼
strip pipeline-internal keys    ← remove _extraction_confidence, _warnings, etc.
        │
        ▼
Model.model_validate(parsed)    ← Pydantic schema + OCL validators
        │
        ├── ValidationError     → REJECTION QUEUE
        │      error messages   → fed back into corrective retry prompt
        │
        └── warnings.warn()     → COMPLIANCE GAP LOG
               (severity=warning OCL constraints)
        │
        ▼
Validated model instance
        │
        ▼
Model repository (one PolicyStatement per article/clause)
```

### Rejection loop

When `model_validate()` raises a `ValidationError`, the error messages are
structured enough to drive an automatic retry:

```python
try:
    instance = LegalBasisModel.model_validate(parsed)
except ValidationError as e:
    retry_prompt = (
        f"Your previous extraction was rejected for these reasons:\n"
        f"{e}\n\n"
        f"Retry using only the following text and correct the errors above:\n"
        f"{legal_text}"
    )
    # Re-call the LLM with system_prompt + retry_prompt
    # Maximum 2 retries before routing to human review queue
```

This loop is self-correcting without human intervention for common errors like
wrong enum values or empty required fields.

## Adding a New Concept

When a new class is added to the metamodel (e.g. `DataRetentionAudit`):

1. Add its enum values to `enums.py` if new controlled vocabularies are needed.
2. Add its Pydantic model to `models.py` with correct field types and OCL validators.
3. Wire it to `PolicyStatement` in `models.py` with the correct multiplicity.
4. Add a `_new_concept_prompt()` builder function to `prompts.py`.
5. Register it in `_CONCEPT_BUILDERS`.
6. Add it to `build_assembler_prompt()` parameters.

The prompt content derives from the schema — the enum grammar is generated
automatically, the OCL hints are written once per constraint, and the few-shot
example is the only part requiring manual authoring.

---

## Adding a New Law

When a new law is ingested (e.g. India DPDPA, Singapore PDPA):

1. No metamodel changes needed if the law's concepts map to existing classes.
2. Update the **cross-law mapping** in the relevant concept prompts
   (Section 2 of each concept prompt, e.g. "Maps to: ... | DPDPA §X").
3. Update the **decision guide** if the law uses surface language not already
   covered (e.g. DPDPA uses "Data Fiduciary" instead of "Data Controller"
   → add a row mapping it to `ActorRole.DataController`).
4. Add a canonical `Jurisdiction` instance for the new law's jurisdiction
   (e.g. `{"jurisdictionId": "IN", "name": "India", ...}`).

The schema itself does not change. The law-specific variation is handled
entirely in the prompts and the RAG retrieval layer.
